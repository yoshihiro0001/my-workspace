## はじめに（この書類の読み方）

この書類は **「何を作るか」「どう動くか」「なぜそう作るか」** を定義した、このプロダクトの **開発の憲法** です。

単なるアプリ開発の指示書ではなく、オーナーの **「美学」と「直感」をシステムに翻訳した設計思想書** として扱ってください。

### テンプレートリポジトリで実装するときの「正」

次のドキュメントと **ずれて実装してはいけません**。

| 種別 | パス（リポジトリルート基準） |
|------|------------------------------|
| **UI・デザイン** | `docs/DESIGN_HANDOVER.md`（ダーク＆ガラス・Tailwind・lucide・Framer Motion） |
| **プロダクト判断の憲法** | `docs/00_constitution.md` |
| **変更履歴（唯一の正）** | `docs/01_history.md`（実装後は必ず追記） |
| **デプロイ・検証** | `docs/02_operations.md` |
| **AI エージェント短則** | `AGENTS.md` |

### 実装スタック（固定）

- **フロント（ワークスペースの殻）**: Vite 6 + React 19 + TypeScript + Tailwind CSS 3
- **アイコン**: `lucide-react`
- **アニメーション**: `framer-motion`（`motion` コンポーネント）
- **本番の静的配信**: `server.js`（Express が `dist/` を `SUBPATH` 配下で配信）

**上記フロントスタックを別物に置き換える判断は行わないでください。** バックエンド・API 連携・ファイルシステム操作・dev サーバー管理等の技術選定はフェーズに応じて自由です。判断は `docs/01_history.md` に記録すること。

### ユーザー体験について

- **ユーザーはコードが読めない非エンジニア** です。
- コードを読めないからこそ **「視覚とロジック」で開発をコントロール** できる体験を実現します。
- 「圧倒的にシンプル・直感的・ユーザーフレンドリー」が最優先。

## 0. 開発の最優先原則（コア・フィロソフィー）

全ての実装判断は、以下の 3 原則に従うこと。迷ったらここに戻る。

### 原則① 非 AI ロジックによる爆速反応

要素の選択、コードのハイライト、数値の微調整に **AI を使ってはならない**。
これらはブラウザのネイティブロジック（ソースマップ、DOM 操作等）で **体感 0.1 秒以内** に実行すること。

> AI は「考える」作業に使う。「反応する」作業はネイティブロジックに任せる。

### 原則② 透明性の確保

AI が「なぜそのコードを書いたか」の **根拠（憲法・履歴・デザインガイド）を常に可視化** する。
ブラックボックスのまま「できました」は禁止。

### 原則③ スマホ・ファースト

PC を開く必要がない。スライダーとタップだけで、**GitHub 上の本物のコードが書き換わる** 体験。

## 1. このアプリが解決する問題

コードが読めない人が、複数のアプリ開発を同時進行したい。しかし今は：

- 開発の全工程を AI に丸投げするしかない
- 途中の状態が **見えない**（本当にできてるか不安）
- 構成説明やモックでは **実感がない**（実際のサイトを見たい）
- 修正が **怖い**（戻せるかわからない）
- AI の判断が **不透明**（なぜその変更をしたか見えない）

**このアプリはその全てを解決する。**

## 3. 絶対に外せない 3 つの核心

### 核心① 本物のサイトを見ながら開発する

「モックを眺める」のではない。  
**実際に動いているサイトそのものをスマホ画面いっぱいに表示し、それを見ながら開発する。**

```
プレビュー = サーバーで動いている本物のサイト
        ↓
要素をタップ → ソースマップで該当コードが瞬時にハイライト（非 AI）
        ↓
スライダーで数値を調整 or AI に指示 → コードが書き換わる
        ↓
プレビューがリアルタイムに更新される（本物のサイトが変わる）
        ↓
「保存」＝ Git コミット。いつでも任意の時点に戻せる
```

構成説明やモックプレビューとは根本的に違う。**開発者が見ている画面と同じ本物を見る。**

### 核心② 触って・光って・調整する（非 AI ロジック）

コードを読めなくても、**「見て」「触って」「光らせて」「調整する」** サイクルが回る。

- **見る**: 本物のサイトがスマホ画面いっぱいに表示される
- **触る**: 気になる要素をタップ → その要素が選択される
- **光る**: 選択した要素に対応するコードが瞬時にネオンハイライト
- **調整する**: 数値（サイズ・色・余白等）をスライダーで左右にスワイプ → リアルタイム反映

この全てが **AI を経由しない**。ブラウザのネイティブロジックで 0.1 秒以内に動く。

### 核心③ AI は「根拠付き」で動く

AI に丸投げではなく、**「絞り込んで渡す」＋「根拠を明示させる」** で精度とコストを最適化。

- 選択中のコードと関連する憲法の一節を自動でプロンプトに結合
- 「憲法の○行目に基づき、このスタイルを適用しました」と根拠を表示
- 重要な決定事項は `AGENTS.md` に自動追記（AI が過去のこだわりを忘れない）

## 4. 物理構造と同期ロジック

このアプリは **「スマホ（リモコン）」** で **「クラウドサーバー（作業部屋）」** を操作する。

### GitHub リアルタイム同期

```
ユーザーがスマホで操作（タップ・スライダー・AI 指示）
        ↓
サーバー上の実ファイルに即時反映
        ↓
dev サーバー（Vite HMR）がリアルタイムにリロード
        ↓
プレビュー（本物のサイト）が更新される
        ↓
キリの良いタイミングで Auto-Commit → GitHub に履歴を刻む
```

- ファイルの書き換えは **サーバー上の実ファイル** に即時反映される
- 「保存」＝ Git commit。**全ての変更が履歴に残り、いつでも戻せる**
- Auto-Commit はユーザーの明示的な「保存」アクション、または一定時間操作がなかったときに実行

### ソースマップ連携

- プレビュー上の DOM 要素と、ソースコードの行番号を **1 対 1 で紐付ける**
- **AI を介さず**、タップした瞬間に該当コード行を特定する
- Vite の開発サーバーが生成するソースマップ、または JSX ソース位置注入プラグインを活用

## 5. 視覚的インターフェース：3 層レイヤー構造

画面を「切り替える」のではなく、**「重ねる」** ことで情報の断絶をなくす。

### ① プレビュー・レイヤー（最前面）

**そのプロジェクトの実際のサイトをフルサイズで表示する。**  
モックやスクリーンショットではない。サーバーで動いている本物のサイトそのもの。

| 機能 | 説明 |
|------|------|
| フルスクリーン表示 | スマホ画面全体を使い、実際のサイトをそのまま表示 |
| インスペクタ・モード | 画面上のあらゆる要素（ボタン、余白、親ボックス）を選択可能 |
| 階層スライダー | 直接触りにくい小さなパーツや親要素を順番にフォーカスするスライダー（非 AI） |
| デバイス切り替え | スマホ / タブレット / PC 表示を切り替え（デフォルトは閲覧デバイスに合わせる） |

### ② ライブコード・レイヤー（下部スライド）

プレビューの下からスライドアップして表示される、対応するコードの表示・編集領域。

| 機能 | 説明 |
|------|------|
| ネオン・ハイライト | プレビューで選んだ要素のコードが瞬時に光る |
| スマート・ボールド | 数値（`12px`）や色コード（`#FFFFFF`）など調整頻度の高い箇所を自動太字表示 |
| 物理スライダー編集 | 太字部分を左右スワイプでコードの数値が直接書き換わり、プレビューがリアルタイムに変化 |
| ファイルパス・行番号 | 今どのファイルの何行目を触っているか常に表示 |

### ③ エージェント・レイヤー（サイド / ポップアップ）

AI への指示と、AI の判断根拠を表示する領域。

| 機能 | 説明 |
|------|------|
| 自然言語入力 | 「この余白をもう少し広くして」のような日本語指示 |
| 憲法ビュー | `docs/00_constitution.md` を常に参照可能 |
| AI 切り替え | Claude / GPT をワンタップで頭脳切り替え |
| 根拠表示 | 「憲法の○行目に基づき、このスタイルを適用しました」 |
| チャット履歴 | 指示と応答を時系列で表示（既存 Phase 7 の資産を発展） |
| 候補チップ | 「余白を広く」「角を丸く」などよく使う指示をワンタップ |

## 6. AI エージェントの動作アルゴリズム

### コンテキストの自動生成

```
ユーザーが選んでいる「光っているコード」（ターゲット）
    ＋
プロジェクト全体マップ（ファイル構成の概要）
    ＋
対象ファイル全文
    ＋
関連する憲法・デザインガイドの該当節
    ↓
自動でプロンプトに結合して AI に送信
```

ユーザーは「選んで」「指示する」だけ。コンテキストの組み立ては全自動。

### 判断根拠の表示

AI が修正案を出した際、**必ず** 以下を明示する：

- **変更箇所**: どのファイルの何行目をどう変更したか
- **根拠**: 「`00_constitution.md` の○行目に基づき」「`DESIGN_HANDOVER.md` の△の規定に従い」
- **影響範囲**: 他に影響するファイルがあればその旨を通知

### メモリ管理

- 重要な決定事項は `AGENTS.md` に自動追記
- AI が「前回のこだわり」を忘れない仕組み
- 履歴（`docs/01_history.md`）を毎回参照し、過去の判断と矛盾しない

## 7. ファイルに対してできる操作（全て）

| 操作 | 説明 |
|------|------|
| プレビュー | 実際のサイトをフルサイズで表示 |
| 要素選択 | タップで要素を選択 → 対応コードがハイライト |
| スライダー編集 | 数値・色をスライダーで直接調整 → リアルタイム反映 |
| AI 編集 | 自然言語で指示 → コード書き換え → プレビュー更新 |
| 保存（コミット） | 変更を確定し GitHub にコミット |
| 戻す | 直前のコミット状態に戻す（1 ステップ Undo） |
| 履歴 | Git 履歴を一覧で見る・任意の時点に戻す |
| 比較 | 2 つのバージョンを並べて差分を確認 |
| 作成 | ファイル / フォルダの新規作成 |
| 名前変更 | ファイル / フォルダの名前を変更 |
| 移動 | 別のフォルダへ移動 |
| 削除 | ゴミ箱へ移動（即削除しない。復元可能） |

## 8. データの考え方

### ファイルの正本

| レイヤー | 役割 |
|----------|------|
| **サーバー上の実ファイル** | 編集対象の正本。dev サーバーがここを参照 |
| **Git（GitHub）** | バージョン管理の正本。全履歴を保持 |
| **ブラウザ（IndexedDB）** | 設定・キャッシュ・UI 状態のみ |

### 移行について

Phase 1〜8 で構築した IndexedDB ベースの仮想ファイルシステムは、サーバー実ファイル連携の完成後に段階的に置き換える。移行期間中は既存機能をレガシーモードとして維持し、動作確認しながら切り替える。

### データ肥大化を防ぐルール

- Git 履歴は GitHub に委ねる（ローカルでの件数制限は不要）
- AI 編集は **対象ファイルのコード＋憲法の関連節のみ** 渡す（プロジェクト全体は渡さない）
- プレビューは **dev サーバーを直接表示** する（HTML を自前で生成しない）

## 9. 既存システムとの関係（本番サーバー）

同じサーバー・同一ドメイン上に **既存アプリ** が動いている場合、**そのアプリのパス・PM2・DB は変更しない**。

本ワークスペースの公開パス（SUBPATH）・内部ポート・配置ディレクトリは、リポジトリの **`deploy.config.sh`**（`npm run bootstrap:derive` の出力）と **Nginx** で **既存と被らない値** にする。

```
サーバー（例）
├── /tax                ← 既存（触らない）
├── /my-workspace       ← ワークスペース本体 UI
└── /my-workspace/api/  ← ファイル操作・Git 連携・dev サーバー管理 API
```

管理対象プロジェクトの dev サーバーは **ワークスペース API が動的に起動・停止** し、内部ポートを自動割り当てする。

## 12. 開発者（AI）への厳命事項

1. **「コードを読めないユーザー」を軽視するな**: エラーメッセージをそのまま出すのは禁止。すべて日本語の「改善提案」に翻訳すること。

2. **「戻せる」安心感**: すべての操作は 1 ステップで Undo 可能にすること。Git のコミット履歴がその基盤。

3. **引き算の美学**: ユーザーが迷うような余計なボタン・メニューは一切排除し、常に「今やるべきこと」だけを提示せよ。

4. **速度への執着**: 要素選択・ハイライト・スライダー操作は AI を経由せず、ネイティブロジックで 0.1 秒以内に反応させよ。

5. **根拠なき変更の禁止**: AI が変更を提案する際、必ず憲法・履歴・デザインガイドのどこに基づく判断かを明示せよ。

6. **本物を見せろ**: プレビューは本物のサイト。モックやスクリーンショットは「見えない → 不安」の根源。

---

## UX で最重要視してほしいこと

| 優先度 | 原則 | 説明 |
|--------|------|------|
| 1 | **実感がある** | プレビューは本物のサイト。構成説明やモックではない |
| 2 | **迷わない** | どこをタップすれば何ができるか、考えなくてもわかる |
| 3 | **怖くない** | 全ての変更は Git で管理。いつでも戻せる |
| 4 | **速い** | タップからハイライトまで体感 0.1 秒以内 |
| 5 | **読まなくていい** | コードもエラーメッセージも、ユーザーの目に触れない |
| 6 | **根拠がある** | AI の判断が透明で、なぜそう変えたかが見える |

## デザイン・UI 実装について

**ワークスペースアプリの画面一式** は **`docs/DESIGN_HANDOVER.md` に従ってください。**

フォント・色・角丸・余白・アニメーション・コンポーネントのトーンは、**そのファイルが唯一の正本** です。

3 層レイヤー構造の新しい UI も、同じダーク＆ガラスモーフィズムのトーンで統一すること。ライブコード・レイヤーのシンタックスハイライトは、ダーク背景に映えるネオン系の配色（エメラルド / シアン / パープル）を基調とする。

## 最後に

このアプリの価値は「コードを書けない人が、**本物のサイトを見ながら**、直感的に開発の全工程をコントロールできる感覚を持てる」ことです。

構成説明やモックでは「本当にできてるか不安」になる。  
だから **実際に動いているサイトそのもの** を見て・触って・調整できるようにする。

**「見る → 触る → 光る → 調整する → 保存する」**  
この 5 ステップが気持ちよく動くことを最優先にしてください。
