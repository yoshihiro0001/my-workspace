「技術選定はすべてお任せ」は **ワークスペースアプリの内部（API・DB・仮想ファイル保存・AI 連携）** に適用します。**上記フロントスタックを別物に置き換える**判断は、このテンプレート方針と矛盾するため行わないでください。

## このアプリが解決する問題

コードが読めない人が、複数のアプリ開発を同時進行したい。
しかし今は、開発の全工程をコーディングAIに丸投げするしかなく、
途中の状態が見えない・管理できない・修正が怖い、という問題がある。

**このアプリはその問題を解決する。**

---

## このアプリの本質（一文で）

> 「スマホから自然言語で指示するだけで、実際のコードが書き換わり、
> その結果をリアルタイムで確認・保存・管理できる、AI搭載のファイルマネージャー。」

---

## 絶対に外せない3つの核心

### 核心① 指示したら、本物のコードが変わる

「プレビューを生成する」のではない。
「実際のコードファイルをAIが書き換え、その結果がプレビューとして表示される」。

```
ユーザーが自然言語で指示
        ↓
AIが対象ファイルのコードを実際に書き換える
        ↓
書き換えたコードがプレビューに表示される（本物）
        ↓
「保存」を押すとファイルに確定保存される
        ↓
「戻す」を押すと直前の状態に戻る
```

これが最重要。モックやプレビュー生成とは根本的に違う。

---

### 核心② ファイルとフォルダで管理する

プロジェクトの全要素を「ファイル」として管理する。
ファイルはフォルダに入り、フォルダはフォルダに入る。階層に上限はない。

以下は **ユーザーが見る論理ツリーの例**（実装は React の状態・DB 上のツリー等にマッピングしてよい）。

```
プロジェクト（例：税金申告アプリ）
├── ページ/
│   ├── トップページ.html
│   ├── ログイン画面.html
│   └── ダッシュボード.html
├── スタイル/
│   └── 共通スタイル.css
├── 機能/
│   ├── ログイン処理.js
│   └── データ取得.js
└── 共通設定/          ← プロジェクト作成時に自動生成
    ├── フォント・色
    ├── ボタンスタイル
    └── アニメーション
```

ユーザーは複数のプロジェクトを同時進行する。
プロジェクトはワークスペースのトップ画面でカード一覧として管理する。

**ワークスペース本体の UI** は `docs/DESIGN_HANDOVER.md`（ダーク＆ガラス）に従う。  
上記ツリー内の「プレビューで動かす中身」はユーザー制作物のため HTML/CSS/JS 等のままでよい。

---

### 核心③ ファイルを開くと3つの顔が見える

1つのファイルを開いたとき、以下の3つを切り替えて見られる。

**顔1「プレビュー」**
実際のコードが動いている画面。スマホ・タブレット・PC表示を切り替えられる。（デフォルトは見ているデバイスに合わせる）

**顔2「構成説明」**

日本語の説明文で記述する。（コードから自動生成）

下表は **説明フィールドの例**。実際のワークスペース画面の見た目は **`docs/DESIGN_HANDOVER.md`（ダーク基調・ガラス）** に統一する。例の色・レイアウトはあくまで **説明文のサンプル** であり、ワークスペース chrome の配色と混同しないこと。

| フィールド | 説明 | 例 |
|---|---|---|
| 名前 | ボックスの名称 |「ヘッダー」「送信ボタン」 |
| 見た目の説明 | どう見えるか | 「ガラスカード上にタイトル、右にアイコン」 |
| フォント | 使用フォント | 「共通フォントを使用」または「Inter 16px 太字」 |
| 色 | 背景・文字・ボーダーの色 | 「DESIGN_HANDOVER のアクセントに準拠」または「エメラルド系で成功表示」 |
| 機能・動作 | ボタンを押したら何が起きるか | 「タップでメニューが下からスライドイン」 |
| アニメーション | 動きの説明 | 「Framer Motion 準拠のフェード、0.25s」 |
| 状態 | 進捗 | 未着手 / 作業中 / 完成 |
| メモ | 補足・懸念点 | 「スマホとPCで表示を変える必要あり」 |

---

**顔3「AI編集」**
自然言語で指示を入力する画面。
指示するとコードが書き換わり、プレビューに即反映される。
バージョン履歴もここから操作する。

---

## ファイルに対してできる操作（全て）

| 操作 | 説明 |
|------|------|
| 開く | ファイルを開き、3つの顔を表示 |
| AI編集 | 自然言語で指示 → コード書き換え → プレビュー更新 |
| 保存 | 書き換えたコードをファイルに確定 |
| 戻す | 直前の保存状態に戻す |
| バージョン履歴 | 過去の保存状態の一覧を見る・任意の時点に戻す |
| 複製 | 同じファイルのコピーを作る |
| 比較 | 2つのバージョンを横並びで見る |
| 移動 | 別のフォルダへ移す |
| 削除 | ゴミ箱へ移動（即削除しない） |
| 復元 | ゴミ箱から元の場所へ戻す |

---

## AI編集の仕組み（開発者向け詳細）

### 入力と出力

```
入力（AIに渡すもの）:
  - 対象ファイルの現在のコード全文
  - ユーザーの自然言語指示（例：「ボタンを青にして」）
  - ファイルの種類（html / css / js 等。ユーザー制作物。ワークスペース本体は tsx 等）

出力（AIが返すもの）:
  - 修正後のコード全文
  - 変更された行の情報（差分ハイライト表示に使う）
```

### ai APIのプロンプト構造

```
システム:
「あなたはWebアプリの開発者です。
ユーザーの指示に従い、提供されたコードを修正してください。
修正後のコード全文のみを返してください。
説明・コメント・マークダウンは不要です。」
```

## データの考え方

### 保存する必要があるもの

- プロジェクト（名前・説明・作成日）
- ファイル・フォルダ（名前・中身・親フォルダ・種類）
- バージョン履歴（ファイルごとに、最新10件）
- ゴミ箱（削除から30日後に自動消去）
- 構成説明のキャッシュ（ファイルのバージョンと紐づけ）

### データ肥大化を防ぐルール

- バージョン履歴は**最新10件のみ**保持。古いものは削除
- AI編集は**対象ファイルのコードのみ**渡す（プロジェクト全体は渡さない）
- 構成説明は**保存時のみ生成・キャッシュ**（開くたびに生成しない）

## 既存システムとの関係（本番サーバー）

同じサーバー・同一ドメイン上に **既存アプリ** が動いている場合、**そのアプリのパス・PM2・DB は変更しない**。

本ワークスペースの公開パス（SUBPATH）・内部ポート・配置ディレクトリは、リポジトリの **`deploy.config.sh`**（`npm run bootstrap:derive` の出力）と **Nginx** で **既存と被らない値** にする。

```
サーバー（例）
├── /tax           ← 既存（触らない）
└── /my-workspace  ← 例: スラッグに合わせたサブパス（実値はプロジェクトごと）
```

## 最高のパートナーコーディング AI へのお願い

### 技術選定について

- **フロント（ワークスペースの壳）**: 上記「実装スタック（本テンプレートで固定）」に従う。  
- **バックエンド・永続化・AI API 連携**: フェーズに応じて選定してよい。`docs/01_history.md` に判断の要約を残すこと。

「このユーザーにとって最もシンプルで使いやすい実装」を選びつつ、**デザイン憲法・リポジトリ憲法と矛盾させないこと**。

### UXで最重要視してほしいこと

1. **迷わない**: どこをタップすれば何ができるか、考えなくてもわかる
2. **怖くない**: 削除・上書きは必ず「戻せる」状態にする
3. **速い**: 指示してからプレビュー更新まで、体感でもたつかない
4. **読まなくていい**: コードもエラーメッセージも、ユーザーの目に触れない

### Phase分けをお願いします

一気に全部作らず、動く状態を段階的に確認しながら進めてください。
「Phase 1が動いたら報告 → 確認 → Phase 2へ」という計画的で最適な進め方を希望します。各フェーズ終了時に **`docs/01_history.md` を更新**すること。
